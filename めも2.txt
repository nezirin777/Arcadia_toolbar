--------------------------------------------------
■ ArcadiaToolBar Next 開発指示書 v2
--------------------------------------------------

対象：
Arcadia 用 Greasemonkey / UserScript

目的：
既存スクリプトを直接延命するのではなく、
新規スクリプトへ段階移植しながら
構造改善を行う。

--------------------------------------------------
■ 基本方針
--------------------------------------------------

既存ファイルを大規模リファクタしない。

理由：

- 暗黙依存が多い
- event順依存が存在する
- table再構築順依存が存在する
- requestAnimationFrame依存が存在する
- DOM構造依存が強い
- Arcadia固有HTML癖が多い

そのため：

「既存コードを仕様書として利用しつつ、
新規構造へ移植する」

方式を採用する。

--------------------------------------------------
■ 開発方式
--------------------------------------------------

新規Userscriptを作成する。

例：

ArcadiaToolBarNext.user.js

または：

ArcadiaToolBar_v4_dev.user.js

--------------------------------------------------
■ 絶対方針
--------------------------------------------------

物理ファイル分割は禁止。

npm禁止。

React禁止。

SPA化禁止。

build前提禁止。

ESModules前提禁止。

Userscript単体配布前提を維持する。

--------------------------------------------------
■ 最重要目標
--------------------------------------------------

1ファイル構成のまま：

- どこに何があるかわかる
- 修正箇所が予測できる
- DOM依存が集中管理される
- 巨大クラス化を防ぐ
- querySelector乱立を防ぐ
- localStorage乱立を防ぐ
- event乱立を防ぐ

状態を構築する。

■ Feature責務
--------------------------------------------------

Featureクラスは：

- orchestration
- init
- event hookup

のみ担当。

--------------------------------------------------

Feature内で：

- 大量querySelector
- regex巨大処理
- localStorage直アクセス
- DOM構築
- renderer直書き
- business logic肥大化

を禁止。

--------------------------------------------------

■ 禁止

■ Renderer方針
--------------------------------------------------

■ Rendererで禁止

--------------------------------------------------
■ Matcher方針
--------------------------------------------------

FavoriteMatcher と NGMatcher を独立させる。

--------------------------------------------------
■ 最重要：部分一致の挙動を維持すること
--------------------------------------------------

お気に入り・NGワードは：

「タイトルへの部分一致」

で動作している。

これは現行の挙動仕様であり、
移植後も必ず維持しなければならない。

--------------------------------------------------
■ 挙動定義
--------------------------------------------------

登録ワード「転生」に対して：

「転生したら魔王だった件」→ マッチする
「転生」→ マッチする
「悪役令嬢」→ マッチしない

この挙動が正とする。

--------------------------------------------------
■ Matcher実装段階
--------------------------------------------------

Phase1:
String.prototype.includes による部分一致

登録件数が少ない間はこれで十分。
実装コストが低く、挙動が明確。

--------------------------------------------------
■ Phase1実装例
--------------------------------------------------

class FavoriteMatcher {
  #items = new Map(); // category -> string[]

  match(title, category) {
    const list = this.#items.get(category) || [];
    return list.some(word => title.includes(word));
  }
}

--------------------------------------------------
■ Phase1の制約
--------------------------------------------------

- 大文字小文字は区別する（現行仕様に合わせる）
- 正規表現は使わない
- Setは使わない（完全一致になるため禁止）

--------------------------------------------------
■ Phase2（将来対応）
--------------------------------------------------

登録件数が大幅増加した場合：

Aho-Corasick アルゴリズムへ移行を検討する。

ただし：

- Phase1で問題が出てから検討する
- 早期最適化禁止
- 移行時も部分一致の挙動を変えない

■ 移植方針
--------------------------------------------------

既存コードは：

「動作仕様書」

--------------------------------------------------
■ 正しい進め方
--------------------------------------------------

- DOM依存を分離

- storage統一
- event整理

を行いながら段階移植する。

--------------------------------------------------
■ 各フェーズ完了基準
--------------------------------------------------

各ステップは以下を満たした時点で完了とする：

- 既存コードと同等の動作が確認できる
- 新旧コードを並走させてdiff確認済み
- 既存コードの該当部分をコメントアウトで
  無効化できる状態になっている

並走期間中は：

既存コードと新コードを同時に動かさない。
（二重初期化・二重イベント登録が起きるため）

切り替えは機能単位で行う。

--------------------------------------------------
■ ListFormatter移植注意点
--------------------------------------------------

ListFormatterは現状：

- parser
- renderer
- matcher
- controller
- storage
- DOM mutation

を抱えている。

最優先で責務分離対象とする。

分離順：

1. StorageManagerへlocalStorage移管
2. FavoriteMatcher / NGMatcherへマッチング移管
3. ArcadiaDOMParserへDOM解析移管
4. ListRendererへDOM更新移管
5. ListFormatter本体をorchestrationのみに削減

--------------------------------------------------
■ 最重要注意点（再掲）
--------------------------------------------------

お気に入り・NGワードの判定は：

部分一致（includes）を維持すること。

Setや完全一致への変更は禁止。

移植後の動作確認項目に必ず含めること。

--------------------------------------------------
■ コード提示ルール
--------------------------------------------------

- コピペしやすい単位で提示
- コメントを適切に記載
- 推測でコードを書かない
- Arcadia実DOMを前提にする
- 部分修正時は関連箇所含めて提示
- 中途半端な差分だけを出さない

--------------------------------------------------
■ 最終目標
--------------------------------------------------

Userscript単体構成のまま：

- 保守可能
- 責務分離済み
- DOM依存集中管理
- 高速
- 事故りにくい

構成を実現する。
